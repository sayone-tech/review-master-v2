Django PasswordResetConfirmView contract (django.contrib.auth.views):
```python
class PasswordResetConfirmView(PasswordContextMixin, FormView):
    form_class = SetPasswordForm        # fields: new_password1, new_password2
    post_reset_login = False
    success_url = reverse_lazy("password_reset_complete")
    template_name = "registration/password_reset_confirm.html"
    token_generator = default_token_generator  # HMAC, single-use via password-hash change

    def form_valid(self, form) -> HttpResponseRedirect:
        user = form.save()
        ...
        return HttpResponseRedirect(self.get_success_url())
    # Template context includes {validlink: bool}. When invalid, renders the
    # same template with validlink=False — templates must branch on {% if validlink %}.
```

Django PasswordResetView template email context (from django.contrib.auth.forms.PasswordResetForm.save):
```
{{ email }}          — recipient email (target user.email)
{{ user }}           — User instance
{{ domain }}         — request domain (e.g. app.reviewmaster.io)
{{ protocol }}       — "http" or "https"
{{ uid }}            — urlsafe-base64 user PK
{{ token }}          — HMAC reset token
{{ site_name }}      — from extra_email_context — we pass "Review Master"

Reset link (to be constructed in template):
{{ protocol }}://{{ domain }}{% url 'password_reset_confirm' uidb64=uid token=token %}
```

In Phase 2 the email is rendered directly by Django's built-in `PasswordResetView`, which calls `render_to_string` on the template and passes the result straight to `send_mail` — there is no `send_transactional_email` wrapper where premailer could hook in yet.
