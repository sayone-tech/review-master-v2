## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gap (`gap-1`), badge dot gap |
| sm | 8px | Compact inline spacing, icon button padding |
| md | 16px | Default form field spacing, modal body padding top/bottom |
| lg | 24px | Modal horizontal padding (`px-6`), section separation |
| xl | 32px | Empty state outer padding vertical |
| 2xl | 48px | Empty state inner padding (`py-[72px]` = 72px vertical, 32px horizontal) |
| 3xl | 64px | Page-level vertical breathing room |

Exceptions:
- Modal header top padding: 24px (`pt-6`) — aligned to spacing scale
- Table cell vertical padding: 16px (`py-4`) — aligned to spacing scale
- Table header vertical padding: 12px (`py-3`) — aligned to spacing scale
- Form input vertical padding: 8px (`py-2`) — aligned to spacing scale

### New React Component: `OrgManagement` entrypoint

Single entrypoint file at `frontend/src/entrypoints/org-management.tsx`. Mounts in `<div id="org-table-root">` inside `organisations/list.html`. Owns:
- DataTable rows
- Row-action three-dot dropdown menu
- All modal dialog state (Create, View, Edit, Enable, Disable, Delete, Store Allocation)

Three-dot row action menu: React-rendered `<div>` positioned absolutely via `useState` toggle. Use `useRef` + `useEffect` for click-outside close. Render via `createPortal` to `document.body` to avoid table overflow clipping. No Alpine.js on this page — React owns all interactivity in the `#org-table-root` mount.

## Modal Specifications

### Create Organisation Modal
- Size: `default` (max-w-[520px])
- Title: "Create Organisation"
- Subtitle: "Fill in the details below. An invitation email will be sent on save."
- Fields (in order):
  1. Organisation Name — text, required, 2–100 chars, placeholder "e.g. Acme Retail Group"
  2. Organisation Type — select, required, options: [("", "Select a type"), ("retail", "Retail"), ("restaurant", "Restaurant"), ("pharmacy", "Pharmacy"), ("supermarket", "Supermarket")]
  3. Email — email, required, unique, placeholder "admin@example.com"
  4. Number of Stores — number, required, min=1 max=1000, placeholder "e.g. 5"
  5. Address — textarea, optional, max 500 chars, placeholder "Full address (optional)"
- Footer: Discard (secondary) | Create Organisation (primary, loading spinner while submitting)
- On success: close modal, dispatch `app:toast` `{ kind: "success", title: "Organisation created. Invitation email sent to {email}.", msg: "" }`

### View Organisation Modal
- Size: `default` (max-w-[520px])
- Title: Organisation name (dynamic)
- Subtitle: none
- Content (read-only detail rows):
  - Type: Badge (neutral)
  - Email: plain text
  - Address: plain text or "—" if empty
  - Stores: "X used of Y allocated"
  - Status: Badge
  - Created: "DD MMM YYYY"
  - Org Admin status: Badge — "Pending invite" (amber dot), "Active" (green dot), "Invitation expired" (gray dot)
  - Last invitation sent: timestamp or "Never"
- Footer: Close (secondary) | "Resend Invitation" (secondary, only when `activation_status !== 'active'`) | "Edit" (primary)
- Detail row pattern: `<dt>` label 12px `text-subtle` uppercase tracking-wide, `<dd>` value 13px `text-text`; use `<dl>` grid `grid-cols-[140px_1fr] gap-x-4 gap-y-3`

### Edit Organisation Modal
- Size: `default` (max-w-[520px])
- Title: "Edit Organisation"
- Subtitle: none
- Fields: Same as Create, all pre-filled; Email field is `disabled` (visually: `bg-line-soft text-subtle`) with helper text "Email cannot be changed after creation."
- Footer: Discard Changes (secondary) | Save Changes (primary, loading spinner while submitting)
- On success: close modal, dispatch `app:toast` `{ kind: "success", title: "Organisation updated.", msg: "" }`

### Disable Organisation Confirmation
- `ConfirmModal` variant: `amber`
- Title: "Disable Organisation"
- Message: "Disable **{org name}**? Disabled organisations cannot be accessed by their Org Admins." (org name in `<strong>`)
- Confirm label: "Disable"
- No `requireTypeToConfirm`
- On success: dispatch `app:toast` `{ kind: "success", title: "{org name} has been disabled.", msg: "" }`

### Enable Organisation Confirmation
- `ConfirmModal` variant: `blue`
- Title: "Enable Organisation"
- Message: "Enable **{org name}**? The Org Admin will regain access immediately." (org name in `<strong>`)
- Confirm label: "Enable"
- No `requireTypeToConfirm`
- On success: dispatch `app:toast` `{ kind: "success", title: "{org name} has been enabled.", msg: "" }`

### Delete Organisation Confirmation
- `ConfirmModal` variant: `red`
- Title: "Delete Organisation"
- Message: "This organisation will be hidden from the list. It can be restored by an administrator."
- `requireTypeToConfirm`: exact organisation name
- Confirm label: "Delete"
- On success: dispatch `app:toast` `{ kind: "success", title: "{org name} has been deleted.", msg: "" }`

### Store Allocation Modal (two-step)
**Step 1 — input:**
- Size: `sm` (max-w-[420px]) — use `Modal` directly (not `ConfirmModal`)
- Title: "Adjust Store Allocation"
- Content: Helper text "Currently using **X** of **Y** stores." Then a number input (min=X) with label "New allocation". Below input, if value < X: amber inline warning "Value cannot be lower than current usage (X stores)." (matches `form_fields.html` `warn` pattern). "Update" button disabled when value < X.
- Footer: Discard (secondary) | Update (primary, disabled when invalid)

**Step 2 — confirmation:**
- `ConfirmModal` variant: `blue`
- Title: "Confirm Store Allocation Change"
- Message: "Change store allocation for **{org name}** from **{old}** to **{new}**?"
- Confirm label: "Confirm"
- On success: dispatch `app:toast` `{ kind: "success", title: "Store allocation updated to {new}.", msg: "" }`

## Interaction Patterns

### Loading States
- **Initial page render:** React reads embedded JSON blob on mount — no spinner needed for first render.
- **Post-mutation table refresh:** React sets `loading=true` on `DataTable` while re-fetching → 6 skeleton rows render with `animate-sk-pulse` animation.
- **Action buttons inside modals:** Show inline spinner (existing `buttons.html` `loading` state: `border-2 border-black/20 border-t-black rounded-full animate-spin`) while DRF call is in-flight. Disable button during loading.
- **Row action menu:** No spinner needed; modal opens immediately, API call happens on modal confirm.

### Error States
- **Form field validation errors:** Rendered inline below the field using `form_fields.html` error pattern (12px red text + `alert-circle` icon).
- **API error on mutation:** Dispatch `app:toast` `{ kind: "error", title: "Something went wrong.", msg: "Please try again. If the problem persists, contact support." }` — do NOT close the modal on error.
- **Email already in use (CORG-02):** Inline field error on Email field: "An organisation with this email already exists."
- **Store allocation below minimum (STOR-02):** Amber inline warning in the input form (not a toast).

### Debounced Search
- Search input in `filter_bar.html` is a GET form submit — no JavaScript debounce needed. The form submits on Enter or user can be given a 400ms debounce via Alpine.js `x-on:input.debounce.400ms="$el.form.submit()"` added to the search input.

### Filter Persistence
- All filter state lives in URL query params (GET form). React reads `window.location.search` on mount and after mutations to preserve active filters in the refresh call.
- After mutation, React calls `GET /api/v1/organisations/?{window.location.search params}` to refresh rows only — no full page reload.

### Focus Management
- All modals use existing `FocusTrap` from `Modal.tsx` — focus traps on open, returns to trigger element on close.
- Delete confirmation: first focusable element is the type-to-confirm input (not the disabled Delete button).
- Row action menu: focus returns to the three-dot button when menu closes.

### Keyboard Navigation
- Three-dot menu: `Space`/`Enter` opens, `Escape` closes, arrow keys move between items.
- Table rows: `Enter` on a clickable Name cell opens View modal.
- All modals: `Escape` closes (dismissible=true on all modals except store allocation step 2 which uses ConfirmModal default).

## Accessibility Contract

All requirements inherited from Phase 1 DSYS-08. Phase 3 specific additions:

- Row action menu must have `role="menu"`, items have `role="menuitem"`, trigger button has `aria-haspopup="true"` and `aria-expanded={open}`.
- Three-dot trigger button aria-label: "Actions for {org name}"
- `DataTable` loading state: table rows have `aria-busy="true"` (already in `DataTable.tsx`).
- Modal dialogs: `role="dialog"` + `aria-modal="true"` + `aria-label` (already in `Modal.tsx`).
- ConfirmModal icon container: `aria-hidden="true"` (already in `ConfirmModal.tsx`).
- Type-to-confirm input: `aria-label="Type organisation name to confirm"`.
- Name column link: `aria-label="View details for {org name}"`.
- Status badges: rendered text is sufficient; no additional aria needed.
- Pagination: existing `aria-label="Pagination"` nav + `aria-current="page"` on active page (already in `pagination.html`).

## Responsive Behaviour

Inherited from Phase 1 DSYS-05 and DSYS-06:
- Desktop (≥1024px): Full table with all columns visible.
- Tablet (768–1023px): Table scrolls horizontally. Sidebar collapses to 64px rail.
- Mobile (<768px): Table transforms to stacked card layout. Sidebar becomes drawer.

Filter bar: stacks vertically on mobile (`flex-col` → `flex-row` at `md:`) — already in `filter_bar.html`.

Modals: full-width with `p-4` container padding on mobile (existing `Modal.tsx` uses `p-6` on container; on mobile this becomes effectively inset from viewport edges).
